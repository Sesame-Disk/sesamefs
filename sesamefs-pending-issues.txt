  =====================================================
  ARCHITECTURE DISCUSSION: Eliminate nginx frontend service?
  =====================================================
  Date: 2026-02-12
  Status: PENDING TEAM DISCUSSION
  Priority: Medium (affects deployment complexity & maintenance burden)

  ISSUE:
  The docker-compose has TWO services serving the same frontend:
    1. "sesamefs" (Go/Gin, port 8082) — ALREADY serves SPA + static files + API
    2. "frontend" (nginx, port 3000) — duplicates SPA + static serving, proxies API to Go

  EVIDENCE THAT GO BACKEND IS SELF-SUFFICIENT:
    - Dockerfile (root) line 71-72: Copies frontend/build INTO Go container
    - server.go line 646-648: router.Static("/static", "./frontend/build/static")
    - server.go line 652-667: NoRoute catch-all serves index.html (SPA) and sysadmin.html
    - sharelink_view.go: Scans frontend/build/static/{js,css} for hashed bundle filenames
    - The Go binary is a fully self-contained monolith — API + SPA + static files

  WHAT NGINX ADDS (marginal benefits):
    - Static asset caching headers: "expires 1y; Cache-Control: public, immutable"
    - Gzip compression (Go server has NO gzip middleware)
    - client_max_body_size 10G for seafhttp uploads
    - Separation of concerns (nginx handles static, Go handles logic)

  PROBLEMS CAUSED BY HAVING NGINX:
    - EVERY new backend route requires ALSO updating nginx.conf proxy rules
      • Example: /u/d/:token was missing from nginx.conf → upload links showed blank on port 3000
      • Example: /d/:token/files/ needed nginx proxy rule too
      • This is a recurring source of bugs — we forget to update nginx.conf
    - Double build: frontend is compiled TWICE (once in root Dockerfile, once in frontend/Dockerfile)
    - Two ports to manage (3000 vs 8082) — confusing for developers
    - nginx.conf must enumerate EVERY API prefix (/api2/, /api/v2.1/, /seafhttp/, /d/, /u/d/, etc.)
    - If a new route doesn't match any nginx location, it falls through to try_files → serves index.html
      instead of proxying to backend → silent failures that are hard to debug

  OPTIONS:

  Option A: REMOVE nginx service entirely (RECOMMENDED for dev, likely for prod too)
    - Use only "sesamefs" container on a single port
    - Add gzip middleware to Go (github.com/gin-contrib/gzip, ~2 lines of code)
    - Add Cache-Control headers for /static/* routes (~5 lines)
    - Go already handles large uploads (no body size limit by default)
    - Eliminates entire class of "forgot to update nginx.conf" bugs
    - Simpler deployment: 1 container, 1 port, 1 binary
    - Reduces Docker build time (no duplicate webpack build)

  Option B: Keep nginx but use generic catch-all proxy (if nginx is required)
    - Change nginx.conf to proxy ALL non-static requests to Go backend:
        location / {
            try_files $uri $uri/ @backend;
        }
        location @backend {
            proxy_pass http://sesamefs:8080;
        }
    - Eliminates need to enumerate every API route
    - Still gets nginx caching/gzip benefits
    - Still adds deployment complexity

  Option C: Keep current architecture (NOT recommended)
    - Continue manually adding proxy rules for every new route
    - Accept recurring bugs when routes are forgotten
    - Accept double webpack build and two port confusion

  RECOMMENDATION:
    Option A for development (immediate).
    Option A or B for production (discuss with team).
    If team wants nginx in production for TLS termination / rate limiting,
    use Option B pattern to avoid route enumeration bugs.
